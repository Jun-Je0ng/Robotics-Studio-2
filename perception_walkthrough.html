What the Perception Subsystem Does

The perception system takes raw camera data and produces structured 3D information about plastic objects — their position, orientation, size, and type — in the robot's coordinate frame, so the robot knows exactly where to reach and how to align its gripper.

The Full Pipeline

RealSense Camera
      │
      ▼
RGB Frame + Depth Frame (aligned)
      │
      ▼
YOLO OBB Model → Oriented Bounding Box (pixel coords + angle + confidence)
      │
      ▼

      │
      ▼

      │
      ▼

What to say: The RealSense D4xx has two sensors — an RGB camera and a depth sensor. Depth pixels don't naturally line up with color pixels because the sensors are physically separated. rs.align(rs.stream.color) warps the depth image so every pixel at position (u,v) in the color image has a corresponding depth value. Without this, the depth you read at a detection's center pixel would be from the wrong physical location.

What to say: We use YOLO with Oriented Bounding Boxes (OBB), not standard axis-aligned boxes. A standard box would waste space and give a wrong rotation for a diagonal bottle. OBB gives us (cx, cy, width, height, angle) — the center pixel, dimensions in pixels, and the rotation angle. We reject detections below 70% confidence to avoid false positives.

Why OBB matters: A regular bounding box around a diagonal bottle would be much larger than the bottle and the angle information would be lost. OBB fits tightly at any angle, which is critical for gripper alignment.

What to say: Given pixel (u, v) and depth d, the 3D camera-frame point is:
  X_cam = (u − cx_principal) × d / fx
  Y_cam = (v − cy_principal) × d / fy
  Z_cam = d

The intrinsics (focal lengths fx, fy and principal point cx, cy) come from the camera's factory calibration, retrieved at startup from the RealSense SDK.

5Camera → Robot Transform lines 76–79

def camera_to_robot(T, cam_xyz):
    p = np.array([cam_xyz[0], cam_xyz[1], cam_xyz[2], 1.0])
    r = T @ p
    return {"x": float(r[0]), "y": float(r[1]), "z": float(r[2])}
What to say: The camera sits above the robot and looks down. The camera has its own coordinate frame (X right, Y down, Z forward into scene). The robot's base_link frame is different. We calibrated a 4×4 homogeneous transform matrix T by placing 4 ArUco markers at known robot-frame positions, detecting them in the camera, and solving for T using least-squares. Now we just multiply: robot_point = T @ camera_point. The 1.0 appended makes it a homogeneous coordinate so translation is included in the matrix multiplication.

7Height Measurement lines 97–139

def compute_height_from_depth(depth_frame, cx, cy, w_px, h_px, depth_object):
    margin = 1.1  # 10% beyond the bounding box edge
    candidates = [
        (int(cx + w_px * margin), cy),   # right of object
        (int(cx - w_px * margin), cy),   # left
        (cx, int(cy + h_px * margin)),   # below
        (cx, int(cy - h_px * margin)),   # above
    ]
    table_depths = [get_median_depth(depth_frame, tx, ty, k=7) for ...]
    depth_table  = float(np.median(table_depths))
    height_mm    = (depth_table - depth_object) * 1000.0
What to say: The camera looks straight down. The table surface is farther from the camera (larger depth value). The object top is closer (smaller depth value). So:
  height = depth_table − depth_object

We sample the table in 4 directions just outside the object's footprint (10% beyond the bounding box edge) and take the median to avoid accidentally sampling the object itself. We validate the result: if height is outside 5–400mm it's rejected as failed. This works for any object orientation because it's purely a vertical measurement.

8Orientation (Angle → Quaternion) lines 81–91

def angle_to_quaternion(angle_rad):
    # Negate: image Y (downward) maps to robot -Y, so angles are mirrored
    r = Rotation.from_euler('z', -angle_rad)
    q = r.as_quat()  # [x, y, z, w]
What to say: YOLO gives us the OBB angle in image coordinates. The image Y-axis points downward, but the robot's Y-axis points forward (away from base). This is a mirror flip. If we didn't negate the angle, the gripper would approach at exactly the wrong angle — perpendicular to the bottle instead of along it. By negating, we convert from image convention to robot convention. We then convert from Euler angle (just rotation around Z) to a quaternion, which is the standard format for 3D orientation in ROS.

Known Limitations

  • One object per class: The system tracks objects by class name. Two bottles of the same type simultaneously will alias — their readings merge rather than being tracked separately.
  • Top-down camera only: Height measurement assumes the camera views the surface from above. Angled views produce incorrect readings.
  • Flat table assumed: The work plane model assumes a roughly flat surface. Significant tilt or curves affect position accuracy.
  • Height below ~8mm unreliable: The RealSense depth noise floor (~3–5mm) approaches the object height for very flat objects.
  • Reflective/transparent objects: Clear bottles and shiny metal cause depth drop-outs. The median kernel mitigates but doesn't fully correct this.
  • Fixed calibration: If the camera is physically moved, the transform matrix must be recalibrated.
  • No occlusion handling: One object hidden behind another will not be detected.

Likely Coach Questions — and How to Answer Them

Q: Why median depth instead of mean?
Median is robust to outliers — zero readings from depth drop-outs or spikes from reflective surfaces. One bad pixel in an 11×11 region doesn't distort the median. Mean would be pulled toward those extremes.
Q: Why 5 stability frames?
At 10Hz, 5 frames = 0.5 seconds. Long enough to reject transient detections (shadows, reflections), short enough not to add meaningful latency to the pick sequence.
Q: Why not use the table-plane model for height?
The table plane model (get_table_z) is calibrated for XY position of the work surface only — it tells you where the table is, not how tall the object sitting on it is. The depth-difference approach directly measures the vertical drop without any additional assumptions.
Q: How accurate is the height measurement?
Typically ±3–5mm for objects taller than ~10mm. Below ~8mm the depth noise floor exceeds the object height, which is why we have the 5mm minimum sanity check — anything below that is reported as failed.
Q: Why OBB and not a standard bounding box?
Standard boxes don't capture rotation. For gripper alignment we need to know which way the long axis of the bottle points to match the gripper fingers to it. OBB gives us that directly as a rotation angle.
Q: Why do you negate the angle in angle_to_quaternion?
The image Y-axis points downward, but the robot Y-axis points forward. This is a mirror reflection. Without negating, the gripper would be rotated 90° from where it should be — perpendicular to the bottle instead of aligned with it.
Q: How was the camera-to-robot calibration done?
We placed 4 ArUco markers at known robot positions measured from the teach pendant. We detected their pixel centres in the camera image. Then we solved for the 4×4 homogeneous transform T (least-squares) that maps those camera coordinates to the known robot coordinates. T is saved as a JSON file and loaded at startup.
Q: What happens if two objects of the same class are present?
Currently the code tracks by class name, so they alias — a known limitation. The system is designed for one object per class on the table at a time.
Q: Why publish JSON on /plastic_detections instead of a typed ROS message directly?
JSON lets us debug the raw output with just "ros2 topic echo" without needing the custom message type installed. The translator node then converts it to the proper typed message for downstream consumption.
